## 3. 構成上の気になる点（候補）

- 各章に「導入 → 本論 → まとめ」のうち、**まとめ（学びの要約/次章への接続）**が明確な節として見当たりません。
  - 章末に要点整理がないため、初学者が「結局何を持ち帰るべきか」で迷いやすい可能性があります。
- 第2〜5章で、章頭に「この章でできるようになること（学習目標）」が明示されていません。
  - 導入文に狙いはありますが、Stage2 の要件（学習目標と節内容の対応）を確認しづらい状態です。
- 第4章の認証・認可（OAuth2.0/JWT 等）は、初学者にとって抽象度が高く情報量も多いため、**本筋（REST + requests）**の理解を阻害するリスクがあります。
- 第5章は Git / 正規表現 / データ構造の3テーマを1章に収めており、章内の話題が切り替わります。
  - 章の狙い（「その他」）は妥当ですが、初学者には「何をどこまでやる章か」が曖昧に感じられる可能性があります。
- 入口文書の整合性（運用上の論点）
  - `README.md` のライセンス表記（MIT）と、`docs/index.md` のライセンス表記（CC BY-NC-SA 4.0）が一致していません。構成そのものではありませんが、読者入口で混乱が起きやすい点です（要確認）。

## 4. 構成改善の提案（案）

### 章共通（全章に適用できる型）

- 各章の先頭に「この章でできるようになること」を追加
  - 例: 3〜5項目の箇条書きで、章内の主要な節と対応が取れるようにする
- 各章の末尾に「まとめ」と「次章へのつなぎ」を追加
  - まとめ: 重要ポイント3〜7個（短文）
  - 次章へのつなぎ: 次章で何ができるようになるかを1〜3文で提示

### 第2章（データ記述言語）

- 章末に `2.4 まとめ`（または `2.x まとめ`）を追加し、JSON/YAML/その他形式の「使い分け基準」を箇条書きで固定化する
  - 初学者向けには「まず覚えるのは JSON/YAML、その他は存在を知る」が読み取りやすいです。

### 第4章（API連携）

- 認証・認可の説明を「必須」と「発展」に分割する
  - 必須: APIキー、Bearerトークン、環境変数での扱い（requestsの実装パターンとセット）
  - 発展: OAuth2.0（複数フローの詳細）、JWT（構造/検証の詳細）
- 章末に「よくあるつまずき（HTTP 401/403/429 など）と確認順」を短く入れると、運用寄り読者の実務接続が良くなります（節追加の候補）。

### 第5章（その他知識）

- 章頭に「この章は3トピックを“最低限”扱う」ことを明示し、到達ラインを定義する
  - Git: clone/add/commit/push/pull と PR ベースの作業イメージ
  - 正規表現: grep/簡単なパターンと、ログからの抽出例
  - データ構造: YAML/JSON のネストを Python で読むときの基本形
- 可能なら、各トピック末尾にミニまとめ（2〜3項目）を入れて、章内の話題切替の負荷を下げる
