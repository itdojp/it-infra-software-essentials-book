# Stage2: 構成レビュー Round1（提案のみ）

## 1. 書籍の概要

- 想定読者（既存記述ベース）
  - IT インフラの運用・構築に携わり、自動化やクラウド連携のためにソフトウェア基礎（設定ファイル、スクリプト、API）を短期間で押さえたいエンジニア。
  - Linux/Windows Server と基本的なネットワーク知識を前提に、DevOps/SRE の実践に向けて開発者との連携を強化したい担当者。
- 本書で達成しようとしているゴール（第1章 1.2 の「到達目標」要約）
  - JSON/YAML などの設定ファイルを理解し、読み書き・編集できる。
  - シェルスクリプト / Python で簡単な自動化スクリプトを作成・実行できる。
  - RESTful API の基本（HTTP メソッド、ステータスコード）を理解し、Python（requests）で API 経由の操作を試せる。
  - Git による設定ファイルやスクリプトのバージョン管理に参加できる。
  - 正規表現の基礎を理解し、ログ解析・文字列処理に活用できる。

## 3. 構成上の気になる点（候補）

- 章ごとの「導入 → 本論 → まとめ」のうち、**「まとめ」相当の節が明示されていない章が多い**（第2〜第5章）。学習内容の定着（要点の再提示、次章への接続）が弱くなりやすい。
- 第1章で「演習問題を豊富に用意」と記述がある一方、各章の中で**演習/チェックポイントが構造として見えにくい**（見出しとして統一されていない可能性）。
- 第2章の「その他のファイル形式（簡易解説）」は、JSON/YAML の実践と性質が異なり、**章の主題（JSON/YAML の習得）から脱線に見える**可能性がある。
- 第5章は Git / 正規表現 / データ構造をまとめて扱っており、**話題の粒度が大きく異なる**ため、初心者が「今どこを学んでいるか」を見失うリスクがある（ツール・概念・基礎 CS が混在）。
- 章ごとの学習目標の書き方が揃っていない（第1章のみ「対象読者/到達目標」を明示）。シリーズ全体の Stage2 要件（章の学習目標と節内容の対応）の観点では、**章単位の目標が見出しとして統一されていると点検しやすい**。

## 4. 構成改善の提案（案）

- 各章の冒頭に、共通フォーマットで「この章でできるようになること（3〜5項目）」を追加する（第1章の「到達目標」形式を横展開）。
- 各章の末尾に「まとめ（要点3〜5）」「次章への接続（なぜ次が必要か）」を追加し、学習曲線を明示する。
- 第1章で言及している演習の位置づけを明確化するため、各章に「演習/チェックポイント」見出しを追加し、最低1つは手を動かす課題を置く（既存コード例の再利用でも可）。
- 第2章の「その他のファイル形式（簡易解説）」は、以下のいずれかに整理する案：
  - 付録へ移動して「必要になったら参照」位置づけにする
  - 章末に「参考：よく見る形式」程度の短い節に縮約する（JSON/YAML の学習を阻害しない分量にする）
- 第5章の扱いは、以下のいずれかを検討する案：
  - 章を分割して「Git」「正規表現」「データ構造」を別章（または付録）にする
  - 分割しない場合でも、章冒頭で「3トピックの目的」と「どの章の内容に効くか（例: Git→第3章/第4章のコード管理）」を明示して混在感を下げる
